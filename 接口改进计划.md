# 知识库系统接口改进计划

## 1. 背景与目标

当前系统已经具备文档上传、文档入库、文档查询、知识块查询和混合检索能力，但接口边界还偏“功能点直连”，没有完全形成面向前端管理台的稳定资源模型。现有前端也已经开始调用 `/documents`、`/documents/{doc_id}/chunks`、`/search` 等接口，但展示层需要更多分页、筛选、统计、状态和错误信息。

本计划基于现有数据模型重新整理接口，目标是：

- 文档管理围绕 `Document` 做完整 CRUD。
- 知识块管理围绕 `KnowledgeChunk` 做完整 CRUD。
- 检索管理围绕查询条件、过滤条件和结果展示，不做 CRUD。
- 健康检查服务于前端状态展示和部署运维。
- 前端可以调整，一切以资源边界清晰、展示体验合理、后端可持续演进为准。

## 2. 现有模型依据

### 2.1 文档模型

`Document` 是文档生命周期和入库状态的核心模型，主要字段包括：

- `doc_id`
- `title`
- `source_type`
- `source_uri`
- `source_hash`
- `category`
- `version`
- `status`
- `parent_doc_id`
- `root_doc_id`
- `ingest_job_id`
- `created_at`
- `updated_at`
- `metadata`

这么设计文档接口的理由：

- 文档是前端管理台的一级资源，用户天然需要新增、查看、编辑、删除、重新入库。
- 现有模型已经包含状态、版本、来源和元数据，足够承载 CRUD。
- `version` 已经服务于增量更新和乐观锁，接口层应显式暴露版本，避免多人或多任务并发更新时覆盖数据。
- `status` 已经包含 `pending`、`processing`、`active`、`failed`、`deleted`，适合支持软删除和前端状态筛选。

### 2.2 知识块模型

`KnowledgeChunk` 是检索和知识治理的核心模型，主要字段包括：

- `chunk_id`
- `doc_id`
- `doc_version`
- `title`
- `content`
- `content_hash`
- `knowledge_type`
- `category`
- `status`
- `index_status`
- `indexed_at`
- `index_error`
- `asset_refs`
- `source_refs`
- `ingest_job_id`
- `metadata`

这么设计知识块接口的理由：

- 知识块是检索命中的最小业务单元，前端需要能查看、编辑、删除、恢复和重建索引。
- `status` 表达知识块业务状态，`index_status` 表达索引状态，两者必须在前端分开展示。
- `source_refs` 和 `asset_refs` 是知识块可信度展示的基础，详情页必须保留来源和资源引用。
- 手动维护知识块可以提升知识库治理能力，适合后期做人工纠错、审核和补充知识。

### 2.3 检索结果模型

`SearchResult` 和 `SearchResultItem` 已经包含：

- `search_id`
- `query`
- `rewritten_query`
- `total_count`
- `results`
- `score`
- `score_components`
- `asset_refs`
- `source_refs`
- `metadata`

这么设计检索接口的理由：

- 检索是动作，不是资源，不应设计成 CRUD。
- 用户关心的是“搜什么、在哪些范围内搜、如何过滤、结果如何展示”。
- 当前检索链路包含 Query Rewrite、向量检索、BM25、RRF 融合和 LLM Rerank，因此接口需要支持普通模式和调试模式。
- 检索过滤条件必须尽量贴合文档和知识块字段，避免前端出现筛选项有值但后端无法解释的情况。

## 3. 接口总体约定

### 3.1 路径版本

新增新版接口前缀：

```text
/api/v1
```

现有接口继续保留：

```text
/health
/upload
/ingest
/search
/documents
```

保留旧接口的理由：

- 现有前端和测试可能已经依赖旧路径，直接替换风险高。
- 新接口可以先服务新版前端页面，稳定后再逐步迁移旧入口。
- 版本化路径能避免后续字段和行为变更破坏外部调用。

### 3.2 统一响应结构

所有新版接口统一返回：

```json
{
  "data": {},
  "meta": {
    "page": 1,
    "page_size": 20,
    "total": 100
  },
  "error": null
}
```

错误响应：

```json
{
  "data": null,
  "meta": null,
  "error": {
    "code": "DOCUMENT_NOT_FOUND",
    "message": "文档不存在",
    "details": {}
  }
}
```

这么做的理由：

- 前端可以用统一方式处理加载、分页、空状态和错误。
- 后端异常不再直接暴露 Python 异常文本，用户提示更稳定。
- `meta` 与业务数据分离，列表和详情接口都可以复用同一响应结构。

### 3.3 列表通用参数

文档和知识块列表统一支持：

| 参数 | 说明 |
|---|---|
| `page` | 页码，从 1 开始 |
| `page_size` | 每页数量 |
| `keyword` | 关键词搜索 |
| `sort` | 排序字段 |
| `order` | `asc` 或 `desc` |
| `created_from` | 创建时间起点 |
| `created_to` | 创建时间终点 |
| `updated_from` | 更新时间起点 |
| `updated_to` | 更新时间终点 |

这么做的理由：

- 文档和知识块都会增长，必须分页。
- 前端表格需要排序和关键词筛选。
- 时间范围是排查入库问题和运营管理时的高频需求。
- 通用参数能降低前后端重复开发成本。

## 4. 健康检查接口改进

### 4.1 接口清单

| 方法 | 路径 | 用途 |
|---|---|---|
| `GET` | `/api/v1/health/live` | 进程存活检查 |
| `GET` | `/api/v1/health/ready` | 服务就绪检查 |
| `GET` | `/api/v1/health/dependencies` | 依赖状态详情 |

### 4.2 返回示例

```json
{
  "data": {
    "status": "degraded",
    "backend": "postgres",
    "checks": {
      "document_repo": "ok",
      "element_repo": "ok",
      "chunk_store": "ok",
      "vector_index": "ok",
      "bm25_index": "ok",
      "embedding": "failed",
      "llm": "unknown",
      "asset_store": "ok"
    }
  },
  "meta": null,
  "error": null
}
```

### 4.3 设计理由

- `live` 只判断进程是否可响应，适合容器存活探针。
- `ready` 判断核心依赖是否可用，适合部署流量切换。
- `dependencies` 展示详细依赖状态，适合前端系统状态页和问题排查。
- 当前 `/health` 只能返回 `ok`，无法区分“服务活着但检索不可用”的情况。

### 4.4 前端展示建议

新增 `/health` 或“系统状态”页面：

- 顶部显示整体状态：正常、降级、不可用。
- 依赖项使用状态标签展示。
- 对失败项展示错误摘要和最近检查时间。

## 5. 文档 CRUD 接口

### 5.1 接口清单

| 方法 | 路径 | 用途 |
|---|---|---|
| `GET` | `/api/v1/documents` | 查询文档列表 |
| `POST` | `/api/v1/documents` | 创建文档记录 |
| `GET` | `/api/v1/documents/{doc_id}` | 查询文档详情 |
| `PATCH` | `/api/v1/documents/{doc_id}` | 更新文档 |
| `DELETE` | `/api/v1/documents/{doc_id}` | 删除文档 |
| `POST` | `/api/v1/documents/{doc_id}/restore` | 恢复文档 |
| `POST` | `/api/v1/documents/{doc_id}/ingest` | 对文档触发入库 |

### 5.2 文档列表

请求示例：

```http
GET /api/v1/documents?keyword=手册&category=设备维护&status=active&source_type=pdf&page=1&page_size=20
```

支持过滤：

| 参数 | 说明 |
|---|---|
| `keyword` | 匹配标题、来源 URI 或元数据 |
| `category` | 文档分类 |
| `status` | 文档状态 |
| `source_type` | 文档类型，例如 `pdf`、`docx`、`xlsx`、`html` |
| `parent_doc_id` | 父文档 ID |
| `root_doc_id` | 根文档 ID |
| `ingest_job_id` | 入库任务 ID |

返回示例：

```json
{
  "data": [
    {
      "doc_id": "doc_xxx",
      "title": "设备维护手册",
      "source_type": "pdf",
      "source_uri": "file:///uploads/manual.pdf",
      "source_hash": "sha256:xxx",
      "category": "设备维护",
      "version": 2,
      "status": "active",
      "parent_doc_id": null,
      "root_doc_id": null,
      "ingest_job_id": "doc_xxx",
      "chunk_count": 36,
      "element_count": 120,
      "asset_count": 8,
      "index_summary": {
        "pending": 0,
        "indexing": 0,
        "indexed": 36,
        "failed": 0
      },
      "created_at": "2026-06-16T10:00:00Z",
      "updated_at": "2026-06-16T10:30:00Z"
    }
  ],
  "meta": {
    "page": 1,
    "page_size": 20,
    "total": 1
  },
  "error": null
}
```

设计理由：

- 前端文档列表需要直接展示统计信息，否则每行都要额外请求详情，性能和体验都会变差。
- `index_summary` 能让用户快速发现索引失败文档。
- 列表返回 `source_uri` 和 `source_hash` 便于排查重复文件和来源问题。

### 5.3 创建文档

请求：

```http
POST /api/v1/documents
```

```json
{
  "title": "设备维护手册",
  "source_type": "pdf",
  "source_uri": "file:///uploads/manual.pdf",
  "source_hash": "sha256:xxx",
  "category": "设备维护",
  "metadata": {
    "department": "售后部",
    "owner": "张三"
  },
  "ingest_after_create": true
}
```

返回：

```json
{
  "data": {
    "doc_id": "doc_xxx",
    "title": "设备维护手册",
    "status": "pending",
    "version": 1,
    "ingest_job_id": "doc_xxx"
  },
  "meta": null,
  "error": null
}
```

设计理由：

- 创建文档和上传文件不应强绑定，方便支持本地文件、MinIO 文件、HTTP 地址和人工登记。
- `ingest_after_create` 让前端可以选择“只登记”或“登记后立即入库”。
- 新建文档默认 `pending`，符合现有生命周期。

### 5.4 查询文档详情

请求：

```http
GET /api/v1/documents/{doc_id}
```

返回应包含：

- 文档基础字段。
- 入库任务信息。
- 统计信息。
- 子文档数量。
- 最近索引状态。
- `metadata`。

设计理由：

- 文档详情页是前端进入知识块、解析元素、资源和入库任务的入口。
- 聚合信息放在详情接口中，可以减少前端多次请求。

### 5.5 更新文档

请求：

```http
PATCH /api/v1/documents/{doc_id}
```

```json
{
  "title": "新版设备维护手册",
  "category": "设备维护",
  "status": "active",
  "metadata": {
    "department": "售后部",
    "owner": "李四"
  },
  "expected_version": 2
}
```

设计理由：

- 文档更新主要用于元数据治理，不应默认重新解析文件。
- 使用 `expected_version` 对齐现有乐观锁设计，防止前端保存旧数据覆盖新数据。
- 如果更新 `source_uri` 或 `source_hash`，应提示用户需要重新入库。

### 5.6 删除与恢复文档

删除请求：

```http
DELETE /api/v1/documents/{doc_id}
```

恢复请求：

```http
POST /api/v1/documents/{doc_id}/restore
```

设计规则：

- 默认软删除，将 `Document.status` 设置为 `deleted`。
- 文档下 `active` 知识块同步设置为 `deleted`。
- 检索索引中的对应 chunk 状态同步为 `deleted`，避免被检索命中。
- 恢复文档时，文档状态恢复为 `active`，知识块可选择一起恢复。

设计理由：

- 现有枚举已经预留 `deleted`，使用软删除比物理删除更符合知识库审计需求。
- 删除文档后检索结果不应继续命中该文档内容，因此必须同步知识块和索引状态。
- 恢复能力可以降低误删成本。

### 5.7 文档入库

请求：

```http
POST /api/v1/documents/{doc_id}/ingest
```

```json
{
  "mode": "incremental",
  "options": {
    "max_depth": 2,
    "max_elements_per_doc": 1000
  }
}
```

设计理由：

- 入库是文档资源上的动作，因此放在文档子路径下比全局 `/ingest` 更符合前端心智。
- `mode` 可表达新建入库、增量更新或强制重建。
- 现有 `/ingest` 可继续保留给批量入库和兼容场景。

## 6. 知识块 CRUD 接口

### 6.1 接口清单

| 方法 | 路径 | 用途 |
|---|---|---|
| `GET` | `/api/v1/chunks` | 查询知识块列表 |
| `POST` | `/api/v1/chunks` | 创建知识块 |
| `GET` | `/api/v1/chunks/{chunk_id}` | 查询知识块详情 |
| `PATCH` | `/api/v1/chunks/{chunk_id}` | 更新知识块 |
| `DELETE` | `/api/v1/chunks/{chunk_id}` | 删除知识块 |
| `POST` | `/api/v1/chunks/{chunk_id}/restore` | 恢复知识块 |
| `POST` | `/api/v1/chunks/{chunk_id}/reindex` | 重建单个知识块索引 |
| `POST` | `/api/v1/chunks/batch` | 批量操作知识块 |
| `POST` | `/api/v1/chunks/batch/reindex` | 批量重建索引 |

### 6.2 知识块列表

请求：

```http
GET /api/v1/chunks?doc_id=doc_xxx&category=设备维护&knowledge_type=procedural&status=active&index_status=indexed&keyword=保养&page=1&page_size=20
```

支持过滤：

| 参数 | 说明 |
|---|---|
| `doc_id` | 所属文档 |
| `doc_version` | 文档版本 |
| `category` | 分类 |
| `knowledge_type` | `declarative`、`relational`、`procedural` |
| `status` | `active`、`superseded`、`deleted` |
| `index_status` | `pending`、`indexing`、`indexed`、`failed` |
| `ingest_job_id` | 入库任务 |
| `keyword` | 匹配标题或内容 |
| `has_assets` | 是否有关联资源 |
| `has_sources` | 是否有来源引用 |

返回：

```json
{
  "data": [
    {
      "chunk_id": "chunk_xxx",
      "doc_id": "doc_xxx",
      "doc_title": "设备维护手册",
      "doc_version": 2,
      "title": "日常保养流程",
      "content_preview": "设备日常保养应按照以下步骤执行...",
      "knowledge_type": "procedural",
      "category": "设备维护",
      "status": "active",
      "index_status": "indexed",
      "index_error": null,
      "asset_count": 2,
      "source_count": 3,
      "indexed_at": "2026-06-16T10:30:00Z"
    }
  ],
  "meta": {
    "page": 1,
    "page_size": 20,
    "total": 1
  },
  "error": null
}
```

设计理由：

- 知识块列表是治理页面，高频操作是筛选、定位异常索引、批量处理。
- 返回 `doc_title` 可以避免前端对每个 chunk 再查文档。
- `content_preview` 适合表格展示，完整内容放详情接口，减少列表负载。
- `asset_count` 和 `source_count` 让用户快速判断知识块是否有证据支撑。

### 6.3 创建知识块

请求：

```http
POST /api/v1/chunks
```

```json
{
  "doc_id": "doc_xxx",
  "doc_version": 2,
  "title": "日常保养流程",
  "content": "设备日常保养应按照以下步骤执行...",
  "knowledge_type": "procedural",
  "category": "设备维护",
  "status": "active",
  "source_refs": [],
  "asset_refs": [],
  "metadata": {
    "manual": true,
    "reviewed": true
  },
  "index_after_create": true
}
```

设计理由：

- 支持人工补充知识是知识库运营的常见需求。
- `metadata.manual = true` 可以区分人工创建和 LLM 抽取。
- 创建后可选择立即索引，避免出现“内容已创建但搜不到”的困惑。

### 6.4 查询知识块详情

请求：

```http
GET /api/v1/chunks/{chunk_id}
```

返回应包含：

- 完整 `content`。
- 所属文档摘要。
- `source_refs` 展开后的解析元素位置。
- `asset_refs` 展开后的资源 URI 和展示信息。
- 索引状态和错误信息。
- `metadata`。

设计理由：

- 知识块详情页需要支持人工审核，必须能看到完整内容、来源和资源。
- 来源引用展开后，用户才能判断知识块是否可信。
- 索引错误直接展示，有助于定位检索不可见问题。

### 6.5 更新知识块

请求：

```http
PATCH /api/v1/chunks/{chunk_id}
```

```json
{
  "title": "日常维护流程",
  "content": "更新后的知识块内容...",
  "knowledge_type": "procedural",
  "category": "设备维护",
  "status": "active",
  "source_refs": [],
  "asset_refs": [],
  "metadata": {
    "reviewed": true
  },
  "reindex": true
}
```

更新规则：

- 更新 `title`、`category`、`status`、`metadata`：可只更新存储和索引元数据。
- 更新 `content`：必须重新计算 `content_hash`。
- 更新 `content`、`knowledge_type`、`source_refs`、`asset_refs`：建议默认重建索引。
- 如果 `reindex = false` 且内容发生变化，应返回校验错误或警告。

设计理由：

- 内容变化会影响向量和 BM25 结果，不重建索引会导致“展示内容”和“检索内容”不一致。
- 元数据变化不一定需要重新生成 embedding，但索引中的过滤字段需要同步。
- 明确规则可以让前端在保存时给出准确提示。

### 6.6 删除与恢复知识块

删除请求：

```http
DELETE /api/v1/chunks/{chunk_id}
```

恢复请求：

```http
POST /api/v1/chunks/{chunk_id}/restore
```

设计规则：

- 默认软删除，将 `KnowledgeChunk.status` 设置为 `deleted`。
- 同步向量索引和 BM25 索引中的 `status` 元数据。
- 恢复后如果索引不存在或过期，应自动设置为 `pending` 或触发重建索引。

设计理由：

- 现有检索索引按 `status == active` 过滤，软删除能自然排除结果。
- 物理删除会损失审计和恢复能力，不适合作为默认行为。
- 恢复时必须考虑索引状态，否则前端会出现“已恢复但搜不到”的问题。

### 6.7 重建索引

单个重建：

```http
POST /api/v1/chunks/{chunk_id}/reindex
```

批量重建：

```http
POST /api/v1/chunks/batch/reindex
```

```json
{
  "chunk_ids": ["chunk_a", "chunk_b"]
}
```

设计理由：

- 当前模型已经有 `index_status`、`indexed_at`、`index_error`，说明索引是需要单独治理的状态。
- 重建索引是修复检索异常的核心操作，应从知识块管理页直接触发。
- 批量重建能处理 embedding、Milvus 或 BM25 故障后的补偿场景。

## 7. 检索接口：围绕条件和过滤

### 7.1 接口清单

| 方法 | 路径 | 用途 |
|---|---|---|
| `POST` | `/api/v1/search` | 标准检索 |
| `POST` | `/api/v1/search/preview` | 快速预览，不做 LLM Rerank |
| `POST` | `/api/v1/search/debug` | 调试检索过程 |
| `GET` | `/api/v1/search/filters` | 获取筛选项 |
| `POST` | `/api/v1/search/feedback` | 提交检索反馈 |

设计理由：

- 检索不是资源，不应提供创建、更新、删除。
- 标准检索面向普通用户，调试检索面向管理员和开发者。
- 筛选项由后端提供，可以避免前端写死分类、文档类型和状态枚举。
- 反馈接口为后续评估、调参和排序优化预留闭环。

### 7.2 标准检索

请求：

```http
POST /api/v1/search
```

```json
{
  "query": "设备如何做日常保养？",
  "top_k": 10,
  "filters": {
    "doc_ids": ["doc_xxx"],
    "categories": ["设备维护"],
    "source_types": ["pdf", "docx"],
    "knowledge_types": ["procedural", "declarative"],
    "doc_status": ["active"],
    "chunk_status": ["active"],
    "index_status": ["indexed"],
    "created_from": "2026-01-01T00:00:00Z",
    "created_to": "2026-06-16T23:59:59Z",
    "has_assets": true
  },
  "options": {
    "rewrite": true,
    "hybrid": true,
    "rerank": true,
    "include_assets": true,
    "include_sources": true,
    "include_score_components": true,
    "highlight": true
  }
}
```

返回：

```json
{
  "data": {
    "search_id": "search_xxx",
    "query": "设备如何做日常保养？",
    "rewritten_query": "设备日常保养流程和注意事项",
    "total_count": 18,
    "results": [
      {
        "rank": 1,
        "chunk_id": "chunk_xxx",
        "doc_id": "doc_xxx",
        "doc_title": "设备维护手册",
        "doc_version": 2,
        "title": "日常保养流程",
        "content": "设备日常保养应按照以下步骤执行...",
        "highlight": "...<mark>日常保养</mark>...",
        "score": 0.92,
        "score_components": {
          "vector": 0.81,
          "bm25": 0.63,
          "rerank": 0.92
        },
        "category": "设备维护",
        "knowledge_type": "procedural",
        "asset_refs": [],
        "source_refs": []
      }
    ]
  },
  "meta": null,
  "error": null
}
```

设计理由：

- `filters` 明确表达检索范围，便于前端过滤面板和后端检索实现对齐。
- `options` 控制展示和检索策略，避免为了不同页面复制多个相似接口。
- `doc_title`、`doc_version`、`highlight` 是前端结果列表必须信息，后端一次返回可以减少额外查询。
- `score_components` 保留向量、BM25、Rerank 分数，便于管理员解释排序。

### 7.3 快速预览检索

请求：

```http
POST /api/v1/search/preview
```

设计规则：

- 使用同样的请求体。
- 默认 `rewrite = false` 或可配置。
- 默认 `rerank = false`。
- 返回融合候选结果。

设计理由：

- LLM Rerank 会增加延迟和成本，预览模式适合前端输入调试或后台快速验证。
- 当 LLM 服务不可用时，预览模式仍然可以提供基础检索能力。

### 7.4 调试检索

请求：

```http
POST /api/v1/search/debug
```

返回应额外包含：

- 原始查询。
- 改写查询。
- 关键词。
- 向量检索候选。
- BM25 候选。
- RRF 融合候选。
- Rerank 结果。
- 应用后的过滤条件。

设计理由：

- 当前检索链路较长，结果不符合预期时需要知道问题发生在哪一层。
- 调试接口可以服务开发、测试和知识库运营，不污染普通用户检索结果。
- 对评估和调参很有价值。

### 7.5 检索筛选项

请求：

```http
GET /api/v1/search/filters
```

返回：

```json
{
  "data": {
    "categories": [
      {
        "value": "设备维护",
        "count": 36
      }
    ],
    "source_types": [
      {
        "value": "pdf",
        "count": 12
      }
    ],
    "knowledge_types": [
      {
        "value": "procedural",
        "label": "流程型",
        "count": 20
      }
    ],
    "doc_statuses": ["active", "processing", "failed"],
    "chunk_statuses": ["active", "superseded", "deleted"],
    "index_statuses": ["pending", "indexing", "indexed", "failed"]
  },
  "meta": null,
  "error": null
}
```

设计理由：

- 前端过滤项应来自后端真实数据，而不是写死。
- 每个筛选项带 `count`，能减少用户选择无结果条件的概率。
- 枚举标签可以由后端统一维护，前端展示更一致。

## 8. 前端改造计划

### 8.1 页面结构

建议前端调整为以下页面：

| 页面 | 用途 |
|---|---|
| `/documents` | 文档列表、创建、编辑、删除、入库 |
| `/documents/{doc_id}` | 文档详情 |
| `/chunks` | 全局知识块管理 |
| `/chunks/{chunk_id}` | 知识块详情和编辑 |
| `/search` | 普通检索 |
| `/search/debug` | 检索调试 |
| `/health` | 系统状态 |

设计理由：

- 文档、知识块、检索是三个不同心智模型，页面上应拆开。
- 文档详情页承接文档相关的知识块、解析元素和资源，不让列表页过重。
- 检索页面向使用者，调试页面向管理员，避免普通用户看到过多技术信息。

### 8.2 文档列表页

主要区域：

- 顶部工具栏：新建文档、上传并入库、刷新。
- 筛选栏：关键词、分类、状态、文档类型、时间范围。
- 表格：标题、类型、分类、状态、版本、知识块数、索引状态、更新时间、操作。
- 行操作：查看、编辑、删除、恢复、入库。

设计理由：

- 文档管理是后台高频页面，表格比卡片更适合密集信息。
- 状态和索引摘要直接展示，可以让用户快速定位异常文档。
- 新建、上传、入库需要在同一工作流中衔接。

### 8.3 文档详情页

建议使用 Tab：

- `概览`：基础字段、统计、入库状态。
- `知识块`：当前文档下的知识块列表。
- `解析元素`：标题、段落、表格、图片等元素。
- `资源`：图片、视频、附件。
- `元数据`：JSON 查看和编辑。

设计理由：

- 文档详情信息多，Tab 可以减少页面滚动和认知负担。
- 知识块、解析元素和资源都依附于文档，放在文档详情页符合层级关系。
- 元数据编辑需要独立区域，避免误改核心字段。

### 8.4 知识块管理页

主要区域：

- 筛选栏：所属文档、分类、知识类型、状态、索引状态、关键词。
- 表格：标题、文档、内容摘要、知识类型、状态、索引状态、更新时间。
- 右侧详情抽屉：完整内容、来源、资源、索引错误、元数据。
- 批量操作：删除、恢复、重建索引。

设计理由：

- 知识块数量通常远多于文档，需要强筛选能力。
- 抽屉可以让用户不离开列表就快速检查和编辑。
- 批量重建索引是治理场景中非常实用的能力。

### 8.5 检索页

布局建议：

- 顶部：搜索框、搜索按钮、模式选择。
- 左侧：过滤条件。
- 中间：结果列表。
- 右侧：选中结果详情。

结果卡片展示：

- 标题。
- 文档名。
- 分类。
- 知识类型。
- 高亮摘要。
- 分数。
- 来源按钮。
- 资源预览按钮。

设计理由：

- 检索的核心是快速输入、过滤和阅读结果。
- 左侧过滤符合搜索产品常见交互。
- 右侧详情可以展示完整知识块、来源和资源，不打断结果浏览。

### 8.6 检索调试页

展示内容：

- 原始查询。
- 改写查询。
- 关键词。
- 过滤条件。
- 向量候选。
- BM25 候选。
- 融合候选。
- Rerank 结果。
- 分数明细。

设计理由：

- 检索调试信息对普通用户噪音很大，但对管理员非常关键。
- 独立页面可以帮助测试检索质量和定位召回问题。

## 9. 后端实现计划

### 9.1 第一阶段：接口底座

任务：

- 新增 `/api/v1` 路由分组。
- 新增统一响应模型。
- 新增统一错误码。
- 新增分页参数模型。
- 保留旧接口兼容。

理由：

- 底座统一后，文档、知识块、检索接口可以按同一规范扩展。
- 兼容旧接口可以降低改造风险。

### 9.2 第二阶段：文档 CRUD

任务：

- 完善 `GET /api/v1/documents` 的分页和筛选。
- 实现 `POST /api/v1/documents`。
- 实现 `GET /api/v1/documents/{doc_id}`。
- 实现 `PATCH /api/v1/documents/{doc_id}`。
- 实现 `DELETE /api/v1/documents/{doc_id}`。
- 实现 `POST /api/v1/documents/{doc_id}/restore`。
- 实现 `POST /api/v1/documents/{doc_id}/ingest`。

理由：

- 文档是知识库的入口资源，先完善文档管理可以立即提升前端管理能力。
- 文档删除和恢复会影响知识块和检索索引，越早建立规则越好。

### 9.3 第三阶段：知识块 CRUD

任务：

- 新增 `/api/v1/chunks` 路由。
- 实现知识块列表、详情、创建、更新、删除、恢复。
- 实现单个和批量重建索引。
- 对内容变更自动计算 `content_hash`。
- 对状态变更同步索引状态。

理由：

- 知识块是检索质量治理的核心。
- 没有知识块 CRUD，前端只能展示抽取结果，不能修正和运营知识库。
- 索引状态必须和知识块状态一致，否则检索结果会出现脏数据。

### 9.4 第四阶段：检索条件与过滤

任务：

- 扩展搜索请求模型，支持 `doc_ids`、`categories`、`source_types`、`knowledge_types`、`doc_status`、`chunk_status`、`index_status`。
- 实现 `/api/v1/search/filters`。
- 实现 `/api/v1/search/preview`。
- 实现 `/api/v1/search/debug`。
- 标准检索结果补充 `doc_id`、`doc_title`、`doc_version`、`highlight`。

理由：

- 检索结果质量不仅取决于算法，也取决于用户能否正确限定范围。
- 前端过滤面板需要后端提供稳定筛选项。
- Debug 接口能降低后续调参与排障成本。

### 9.5 第五阶段：前端重构

任务：

- 改造 API 客户端，新增 `/api/v1` 方法。
- 重做文档管理页。
- 新增文档详情页。
- 新增知识块管理页。
- 改造检索页过滤面板和结果详情。
- 新增系统状态页。

理由：

- 后端接口只有被前端合理使用，才真正形成产品能力。
- 新版前端应围绕资源管理和检索体验组织，不再只是简单调用接口。

## 10. 存储层补充建议

### 10.1 文档仓储

建议补充能力：

- 分页查询。
- 关键词查询。
- 按 `source_type` 查询。
- 软删除和恢复方法。
- 聚合统计查询。

理由：

- 当前 `DocumentRepository.list` 只支持少量精确过滤，前端列表能力不够。
- 聚合统计如果在接口层逐条计算，性能会随数据量变差。

### 10.2 知识块仓储

建议补充能力：

- 分页查询。
- 多条件过滤。
- 关键词查询。
- 按 `chunk_id` 批量更新状态。
- 按 `doc_id` 批量更新状态。
- 内容更新后自动维护 `content_hash`。

理由：

- 知识块数量会远多于文档，必须避免全量加载后内存过滤。
- 批量状态更新是删除文档、恢复文档、批量治理的基础能力。

### 10.3 索引同步

建议建立统一方法：

```text
sync_chunk_index_metadata(chunk_ids, status, category, knowledge_type)
```

理由：

- 文档和知识块状态变化都会影响检索可见性。
- 向量索引和 BM25 索引都需要同步元数据，散落在多个接口里容易漏。

## 11. 风险与处理

| 风险 | 影响 | 处理 |
|---|---|---|
| 旧接口和新接口并存导致维护成本增加 | 中 | 新接口稳定后逐步迁移前端，最后废弃旧接口 |
| 删除文档后索引仍可命中 | 高 | 删除时同步更新 chunk 和索引状态 |
| 修改知识块内容后未重建索引 | 高 | 后端强制校验，内容变化默认 reindex |
| 列表接口性能下降 | 中 | 仓储层实现分页和过滤，避免内存过滤 |
| 前端筛选项与后端不一致 | 中 | 使用 `/api/v1/search/filters` 统一提供 |
| 手动创建知识块破坏来源可信度 | 中 | 使用 `metadata.manual` 标记，并允许来源为空但前端提示 |

## 13. 最终边界定义

文档接口的边界：

- 管理文件来源、文档元数据、文档状态、文档版本和入库动作。
- 文档是一级资源，必须支持完整 CRUD。

知识块接口的边界：

- 管理可检索知识单元、内容、分类、状态、来源、资源和索引状态。
- 知识块是一级治理资源，必须支持完整 CRUD。

检索接口的边界：

- 根据查询文本、过滤条件和检索选项返回排序结果。
- 检索不是资源，不做 CRUD。
- 检索接口重点是条件、过滤、展示和调试。

健康检查接口的边界：

- 展示服务和依赖是否可用。
- 不承担业务管理能力。

这样的划分能让后端资源模型、前端页面结构和用户操作心智保持一致：文档负责生命周期，知识块负责知识治理，检索负责找到答案，健康检查负责说明系统当前是否可靠。
