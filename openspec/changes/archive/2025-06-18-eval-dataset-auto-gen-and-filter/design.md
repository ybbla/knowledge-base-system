## Context

### 背景
知识库系统当前的评测体系存在以下痛点：
1. **人工维护成本高**：评测数据需要人工编写，增长缓慢，覆盖率有限
2. **验证反馈慢**：每次修改后需要运行全量评测（60 秒+）才能验证效果
3. **无法聚焦验证**：修复某个问题后难以只验证相关用例
4. **结果不可追溯**：评测结果仅在控制台输出，无法对比历史趋势

### 当前架构
- 评测数据存储在单个 `tests/evaluation/eval_dataset.json` 文件中
- 评测指标包括 Recall@5、MRR、Keyword Recall@5
- 通过 pytest 集成，运行 `tests/evaluation/test_evaluation.py`
- LLM 客户端已存在，可复用火山引擎豆包接口

### 约束条件
- 必须保持向后兼容：现有命令、现有数据格式不被破坏
- 自动生成不得阻塞入库主流程
- LLM 调用失败不能影响文档入库结果

## Goals / Non-Goals

**Goals:**
1. 实现文档入库自动生成 3-5 条评测查询
2. 提供多维度筛选能力，支持快速、精准的评测
3. 实现评测结果持久化和历史对比功能
4. 保持 100% 向后兼容现有评测体系

**Non-Goals:**
1. 不实现自动运行全量评测（评测时机完全由用户控制）
2. 不实现在线管理界面（命令行工具优先）
3. 不实现评测数据人工编辑功能
4. 不实现检索参数自动调优

## Decisions

### 1. 存储方案：分文档存储 + 全局合并
**决策**：
- 分文档数据存储在 `tests/evaluation/datasets/doc_{doc_id}_{date}.json`
- 全局数据集保持 `tests/evaluation/eval_dataset.json`
- 评测结果存储在 `tests/evaluation/results/eval_result_{timestamp}.json`
- 维护 `latest.json` 作为最新结果的快捷引用

**理由**：
- ✅ 按文档存储便于追溯和清理
- ✅ 全局合并文件便于全量评测，兼容现有逻辑
- ✅ 文件系统存储，无需额外数据库依赖
- ✅ Git 可版本管理，便于回滚

**备选方案**：
- 单文件存储：难以追溯来源，合并时易冲突
- SQLite 存储：增加依赖，Git 版本管理不友好

### 2. 触发方式：FastAPI BackgroundTasks
**决策**：使用 FastAPI 内置的 BackgroundTasks，入库完成后异步触发生成。

**理由**：
- ✅ 零额外依赖，复用现有框架能力
- ✅ 不阻塞主入库流程，用户无感知
- ✅ 失败不影响主流程，符合设计要求

**备选方案**：
- Celery/RQ 任务队列：增加基础设施复杂度，本场景不需要持久化任务队列
- 同步生成：阻塞入库流程，体验差

### 3. 筛选架构：Filter 类 + 构建器模式
**决策**：
- 独立 `DatasetFilter` 类封装所有筛选逻辑
- `FilterCriteria` 数据类承载筛选参数
- 函数式筛选器链：每个条件是一个 `(EvalItem) -> bool` 的谓词函数

**理由**：
- ✅ 单一职责原则：筛选逻辑与评测运行逻辑分离
- ✅ 可扩展：新增筛选条件只需添加新的谓词函数
- ✅ 可测试：筛选逻辑可独立单元测试
- ✅ 组合灵活：多个条件自然通过 AND 组合

### 4. 元数据字段默认值策略
**决策**：所有新增元数据字段都有合理默认值，确保旧数据兼容。
- `difficulty`: 默认 "medium"
- `source`: 默认 "auto"
- `category`: 默认 "未分类"
- 时间相关字段：默认 None

**理由**：
- ✅ 保证向后兼容：旧数据不需要迁移即可使用
- ✅ 渐进式增强：后续可逐步完善元数据标注

### 5. LLM Prompt 设计
**决策**：
- System prompt：设定角色、输出格式要求、生成原则
- User prompt：包含文档 chunk 列表 + 目标查询数量 + 格式要求
- 输出 schema：严格 JSON 格式，强制 items 数组结构

**理由**：
- ✅ 结构化输出便于解析
- ✅ 可复用现有 llm_client.chat_json 能力
- ✅ 温度设置为 0.7，保证多样性同时控制质量

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| LLM 生成的查询质量不稳定 | 评测数据噪音多 | 1. 人工审查后才合入全局<br>2. 提供 `--source manual` 筛选，可只跑可信数据 |
| LLM 编造不存在的 chunk ID | 评测指标失真 | 生成后校验 chunk ID 存在性，过滤非法条目 |
| LLM 提取的关键词不在正文中 | 关键词召回指标无意义 | 生成后校验关键词存在性，自动过滤或重新提取 |
| 生成失败导致入库流程异常 | 用户体验受影响 | 所有异常 try-catch，记录日志但不影响主流程 |
| 评测数据文件过多，加载变慢 | 评测启动延迟 | 1. 按需加载，筛选后只加载匹配的<br>2. 考虑缓存机制 |
| 去重逻辑不完善，重复条目积累 | 数据集冗余膨胀 | 1. 按 query 精确匹配去重<br>2. 定期维护工具清理重复 |

**性能考量**：
- 加载 1000 条评测数据 ≈ 10ms，可接受
- LLM 生成每篇文档数据 ≈ 3-5 秒，异步执行不影响用户
- 筛选操作是纯内存计算，时间复杂度 O(n)，可忽略

## Migration Plan

### 部署步骤
1. **Phase 1**：部署存储层和筛选器代码（无行为变化）
2. **Phase 2**：部署评测脚本增强，新增命令行参数（可选功能）
3. **Phase 3**：入库流程集成自动生成（可通过配置开关控制）
4. **Phase 4**：验证数据生成质量，开启默认自动生成

### 回滚策略
- **配置级回滚**：设置 `auto_eval_enabled=false` 即可关闭自动生成
- **代码级回滚**：新功能都是增量代码，不影响现有逻辑，回滚代码即可
- **数据级回滚**：删除 `datasets/` 目录下的自动生成文件，不影响原有 `eval_dataset.json`

### 数据迁移
- 无需数据迁移：旧数据格式完全兼容
- 新生成的数据采用新格式，与旧数据共存

## Open Questions

1. **是否需要 LLM 生成后人工审核？**
   - 当前设计：直接自动合入，提供来源筛选
   - 备选：生成后进入待审核队列，人工确认后合入

2. **是否需要支持批量重新生成所有文档的评测数据？**
   - 当前设计：只支持新入库时自动生成
   - 备选：提供命令行工具，批量为已有文档生成

3. **是否需要实现语义去重（而不是精确文本匹配）？**
   - 当前设计：精确文本匹配去重
   - 备选：Embedding 相似度去重（需要额外 LLM 调用，增加复杂度）

4. **失败的生成任务是否需要重试机制？**
   - 当前设计：失败即放弃，记录日志
   - 备选：简单重试 2-3 次
