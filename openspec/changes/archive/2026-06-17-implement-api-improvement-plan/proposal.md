## Why

当前 API 已支持上传、入库、文档查询、知识块查询和混合检索，但整体仍偏功能点直连，缺少面向前端管理台的稳定资源边界。随着文档、知识块和检索调试需求增加，需要将接口整理为可分页、可筛选、可治理、可解释的新版 `/api/v1` 契约。

本变更解决三个问题：文档缺少完整 CRUD 和状态治理，知识块缺少独立 CRUD 与索引治理，检索过滤能力不足且前端难以解释结果排序。同时保留现有接口兼容，降低迁移风险。

## What Changes

- 新增 `/api/v1` API 分组，并提供统一响应结构：`data`、`meta`、`error`。
- 新增文档管理 API，围绕 `Document` 支持列表、创建、详情、更新、软删除、恢复和触发入库。
- 新增知识块管理 API，围绕 `KnowledgeChunk` 支持列表、创建、详情、更新、软删除、恢复、单个重建索引和批量重建索引。
- 新增系统健康检查 API，区分进程存活、服务就绪和依赖状态详情。
- 扩展检索 API 的过滤和展示能力，支持按文档、分类、来源类型、知识类型、文档状态、知识块状态、索引状态和时间范围过滤。
- 新增检索预览与调试接口，用于低成本快速检索和展示 Query Rewrite、向量候选、BM25 候选、融合候选、Rerank 结果。
- 新增前端页面组织建议：文档管理、文档详情、知识块管理、普通检索、检索调试、系统状态。
- 保留现有 `/health`、`/upload`、`/ingest`、`/search`、`/documents` 接口，不做破坏性删除。

## Capabilities

### New Capabilities

- `document-management-api`: 定义文档 CRUD、文档状态治理、文档入库动作和文档列表展示所需统计信息。
- `chunk-management-api`: 定义知识块 CRUD、知识块状态治理、来源/资源引用展示、索引状态治理和重建索引动作。
- `system-health-api`: 定义存活检查、就绪检查和依赖状态检查，服务前端系统状态页和部署运维。

### Modified Capabilities

- `hybrid-retrieval`: 扩展检索过滤条件、预览检索、调试检索、筛选项接口和结果展示字段。

## Impact

- 影响后端 API 路由：新增 `/api/v1/documents`、`/api/v1/chunks`、`/api/v1/search`、`/api/v1/health`。
- 影响数据访问层：需要补充分页查询、多条件过滤、关键词查询、软删除/恢复、批量状态更新和聚合统计能力。
- 影响索引同步：知识块内容、状态、分类或知识类型变化时，需要同步向量索引和 BM25 索引元数据；内容变化时需要重新索引。
- 影响前端：需要调整为文档管理、文档详情、知识块管理、检索、检索调试和系统状态页面。
- 影响测试：需要新增 API 契约测试、CRUD 行为测试、索引状态同步测试和检索过滤测试。
- 对现有功能的影响：旧接口继续保留，新接口并行提供；现有上传、入库和基础检索流程不应被破坏。
- 回滚计划：若新版接口或前端改造出现问题，可停止挂载 `/api/v1` 新路由或让前端回退到现有接口；数据库层采用软删除和新增字段/查询能力优先，避免不可逆迁移。
