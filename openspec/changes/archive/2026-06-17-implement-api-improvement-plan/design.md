## Context

当前系统的核心数据模型已经形成：`Document` 表达文档生命周期，`ParsedElement` 表达解析结构，`KnowledgeChunk` 表达可检索知识单元，`SearchResult` 表达混合检索结果。现有 API 已提供 `/upload`、`/ingest`、`/search`、`/documents`，但接口形态仍偏单点能力，前端需要自行拼装文档状态、知识块状态、索引状态和检索过滤条件。

本变更将新增 `/api/v1` 接口层，围绕资源边界重新组织能力：文档和知识块作为可管理资源提供 CRUD，检索作为查询动作提供条件、过滤、预览和调试，健康检查作为运维能力提供存活、就绪和依赖状态。

受影响模块：

- `knowledge_base_system/app/main.py`
- `knowledge_base_system/app/api/`
- `knowledge_base_system/app/core/models.py`
- `knowledge_base_system/app/core/deps.py`
- `knowledge_base_system/app/db/repositories/documents.py`
- `knowledge_base_system/app/db/repositories/chunks.py`
- `knowledge_base_system/retrieval/pipeline.py`
- `knowledge_base_system/indexing/`
- `frontend/js/api.js`
- `frontend/js/components/`

## Goals / Non-Goals

**Goals:**

- 新增向后兼容的 `/api/v1` 接口，不破坏旧接口。
- 文档管理支持列表、创建、详情、更新、软删除、恢复和触发入库。
- 知识块管理支持列表、创建、详情、更新、软删除、恢复和重建索引。
- 检索支持更完整的过滤条件，并提供普通、预览和调试三类入口。
- 新增统一响应和错误结构，方便前端处理加载、分页、空状态和错误。
- 前端按照文档管理、知识块管理、检索、检索调试、系统状态重新组织页面。

**Non-Goals:**

- 不移除现有 `/health`、`/upload`、`/ingest`、`/search`、`/documents` 接口。
- 不在本变更中引入新的外部数据库或搜索引擎。
- 不改变现有文档解析和语义抽取算法。
- 不实现权限、审计日志和多租户隔离；这些可在后续独立变更中处理。

## Decisions

### 决策 1：新增 `/api/v1`，而不是直接替换旧接口

理由：旧前端和测试已经依赖现有路径，直接替换会带来不必要的回归风险。新版接口可以先并行提供，前端逐页迁移，稳定后再考虑废弃旧接口。

备选方案：原地扩展旧接口。该方案实现更快，但旧接口响应结构不统一，长期会继续放大前端兼容分支。

### 决策 2：统一响应结构为 `data`、`meta`、`error`

理由：文档和知识块列表需要分页元信息，详情和动作接口需要稳定错误结构。统一响应可以让前端用同一套状态处理逻辑展示加载、错误、空结果和分页。

备选方案：保持 FastAPI 默认返回。该方案代码少，但前端需要为每个接口写不同适配逻辑。

### 决策 3：文档和知识块删除默认使用软删除

理由：现有模型已经预留 `deleted` 状态，知识库系统需要保留审计、恢复和误删补救能力。软删除还能通过索引元数据过滤避免被检索命中。

备选方案：物理删除记录和索引。该方案清理更彻底，但恢复困难，且容易破坏来源追踪。

### 决策 4：知识块内容变化时必须重建索引

理由：`content` 是 embedding 和 BM25 的来源。若内容变化但索引不更新，会出现展示内容与检索命中内容不一致的问题。后端应在内容变化时重新计算 `content_hash`，并强制或默认触发重建索引。

备选方案：允许前端控制是否重建索引。该方案灵活，但容易产生脏索引；仅适合高级维护工具，不适合默认接口。

### 决策 5：检索接口只表达查询、过滤和展示选项，不做 CRUD

理由：检索是动作，不是可持久管理资源。将检索与文档、知识块管理分开，可以避免 API 边界混乱。检索历史和反馈可作为后续独立资源建模。

备选方案：把搜索记录做成完整资源。该方案对日志分析有价值，但不是当前接口改进的核心。

### 决策 6：检索过滤条件以知识块字段为主，必要时关联文档字段

理由：最终被检索和排序的是 `KnowledgeChunk`。`chunk_status`、`index_status`、`knowledge_type`、`category` 应直接作用于候选知识块；`doc_ids`、`source_types`、`doc_status` 需要通过文档元数据或仓储查询转换为候选范围。

备选方案：所有过滤都只放在文档层。该方案前端易理解，但无法表达索引状态、知识类型等真正影响检索结果的条件。

### 决策 7：前端采用管理台式表格和详情抽屉

理由：文档和知识块数量会持续增长，表格、筛选栏、批量操作和详情抽屉更适合高频治理场景。检索页则保持搜索框、过滤面板、结果列表和详情区的查询体验。

备选方案：全部使用卡片展示。该方案视觉轻松，但不适合密集状态、批量操作和运营治理。

## Risks / Trade-offs

- 新旧接口并存增加维护成本 → 通过 API 客户端集中封装和逐步迁移前端降低成本。
- 软删除后索引状态未同步会导致已删内容仍被命中 → 删除和恢复接口必须调用统一索引元数据同步方法。
- 知识块列表数据量大导致慢查询 → 仓储层必须实现数据库分页和过滤，禁止全量加载后内存过滤。
- 检索过滤条件过多导致实现复杂 → 第一阶段先落地 `doc_ids`、`categories`、`knowledge_types`、`chunk_status`、`index_status`，再扩展来源类型和时间范围。
- 调试接口可能暴露内部提示词或敏感错误 → 调试响应只返回检索链路数据、候选 ID、分数和错误摘要，不返回密钥、完整提示词或底层堆栈。

## Migration Plan

1. 新增 `/api/v1` 路由和统一响应模型，旧接口保持不变。
2. 补充文档和知识块仓储分页、过滤、软删除、恢复和统计能力。
3. 实现文档 CRUD，再实现知识块 CRUD。
4. 实现索引状态同步和重建索引动作。
5. 扩展检索 pipeline 的过滤参数，新增 preview、debug 和 filters 接口。
6. 改造前端 API 客户端，并逐步迁移文档页、知识块页、检索页和系统状态页。
7. 新增 API 契约测试和前端基础交互验证。

回滚策略：

- 若新版接口不稳定，可暂时不挂载 `/api/v1` 或让前端切回旧接口。
- 文档和知识块删除使用软删除，回滚时可以恢复状态。
- 若索引同步异常，可通过批量重建索引接口补偿。

## Open Questions

- 是否需要为手动创建的知识块增加审核状态，还是先用 `metadata.manual` 和 `metadata.reviewed` 表达。
- 是否需要在第一阶段实现物理删除，还是长期只保留软删除。
- 检索反馈是否立即持久化为独立表，还是先保留接口占位。
