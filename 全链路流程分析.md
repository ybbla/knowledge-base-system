### 1.2 线程池管理：thread_pool.py

[thread_pool.py:1-660](knowledge_base_system/app/utils/thread_pool.py#L1-L660)

模块提供 `create_thread_pool()` 工厂 + `TaskQueue` 任务队列封装。共 5 个全局线程池，均在 [deps.py:10-22](knowledge_base_system/app/core/deps.py#L10-L22) 模块级初始化（先于 pipeline 等消费模块导入，避免 from-import 绑定 None）：

```
lifespan startup（deps.py 模块加载时触发）:
├── startup_health_pool(max_workers=4, prefix="kb-health")     [441-463]
│   └── 供 /health 端点四路并行探测使用
├── startup_search_pool(max_workers=8, prefix="kb-search")     [582-603]
│   └── 供每次搜索隔离并发（外层），pipeline 内部自建 2 线程池跑双路检索
├── startup_upload_pool(max_workers=8, prefix="kb-upload")     [486-508]
│   └── 供 upload_document 热路径使用（MinIO 写入 + 入队，不阻塞事件循环）
├── startup_asset_worker_pool(max_workers=6, prefix="kb-asset-worker") [535-556]
│   └── 供 _prepare_assets 六路并发（image/video/link 处理）
└── startup_eval_gen_pool(max_workers=8, prefix="kb-eval-gen") [627-648]
    └── 供入库后异步评测数据生成

lifespan shutdown（main.py lifespan yield 后，按序关闭）:
├── shutdown_resources()        → milvus_manager.disconnect()
├── shutdown_health_pool()      → health_executor.shutdown(wait=True)
├── shutdown_search_pool()      → search_executor.shutdown(wait=True)
├── shutdown_asset_worker_pool()→ asset_worker_pool.shutdown(wait=True)
├── shutdown_upload_pool()      → upload_executor.shutdown(wait=True)
└── shutdown_eval_gen_pool()    → eval_gen_pool.shutdown(wait=True)
```

> **注意**：shutdown 顺序为 resources → health → search → asset_worker → upload → eval_gen（先关外部依赖再关业务线程池）。

### 3.2 POST /documents/upload — 上传文档

[documents.py:326-560](knowledge_base_system/app/api/v1/documents.py#L326-L560) — `upload_documents()` + `_batch_upload_sync()` + `_do_upload_sync()`

**重大变更**：支持多文件上传，入库改为异步（Dramatiq Worker），HTTP 请求立即返回 job_id。

```
upload_documents(files[], category, replace_doc_id, confirm_replace)
│
├── [1] 逐个验证（349-383行）
│   ├── 检查文件大小 ≤ max_upload_size_mb
│   ├── 计算 source_hash = sha256(content)
│   └── 重复内容检测 → find_by_hash 命中则标记 duplicate 跳过
│
├── [2] 单文件替换流程校验（373-381行）
│   └── 条件: is_single AND replace_doc_id AND confirm_replace
│       ├── document_repo.get(replace_doc_id)
│       │   ├── 不存在 → 404
│       │   └── status != active → 409 "状态不是 active，无法替换"
│       └── 沿用旧文档的 title/category
│
├── [3] 批量写入 MinIO + 入队（391行）
│   └── loop.run_in_executor(upload_executor, _batch_upload_sync, ...)
│       └── 在线程池中逐文件串行执行 _do_upload_sync
│
└── _do_upload_sync() 同步阶段（424-560行）:
    │
    ├── [3a] 创建 Document 预占位（440-460行）
    │   ├── new_id("doc") → pre_doc_id
    │   ├── Document(source_uri="" 先为空，等文件写完后更新)
    │   │   └── previous_doc_id = replace_doc_id（替换场景记录来源）
    │   ├── document_repo.create(doc)
    │   │   └── DuplicateDocumentError → 409 冲突
    │   └── 设计意图：先占位再写文件，防止并发重复上传产生孤儿文件
    │
    ├── [3b] 写文件到 MinIO（462-485行）
    │   └── upload_api.save_upload_file(file_content, original_name, ...)
    │       ├── 上传到 MinIO kb-input bucket → 返回 source_uri
    │       └── 失败 → document_repo.hard_delete(pre_doc_id) 回滚预占位 → 500
    │
    ├── [3c] 更新 source_uri（487-493行）
    │   └── doc.source_uri = upload_data["source_uri"]
    │       └── document_repo.update(doc) → 将真实 MinIO 路径写回 PG
    │
    ├── [3d] 软删除旧文档（替换场景）（496-498行）
    │   └── 条件: old_doc 非空 → _cleanup_old_doc(old_doc.doc_id)
    │       ├── document_repo.soft_delete(old_doc_id)
    │       ├── chunk_store.bulk_update_status_by_doc_id(old_doc_id, "deleted")
    │       └── sync_index_metadata_batch(old_chunks, ...) → 批量同步 Milvus
    │
    ├── [3e] 创建 IngestJob + 入队 Dramatiq（500-547行）  ← 重大变更：不再同步 ingest
    │   ├── new_id("job") → job_id
    │   ├── job_repo.create(IngestJob(job_id, doc_id, stage="", progress=0))
    │   ├── ingest_document.send(job_id, pre_doc_id) → Dramatiq 消息入队 Redis
    │   ├── 成功 → job.dramatiq_message_id = message.message_id
    │   └── Dramatiq 入队失败 → 全量回滚（删除 job + doc，用户可原样重传）
    │
    └── [3f] 返回响应（549-560行）
        └── 包含 job_id、duplicate、replaced、replaced_doc_id 等标记
```

> **重大变更**：与旧版不同，`ingestion_pipeline.ingest()` 不再在 HTTP 请求中同步调用，而是由 Dramatiq Worker 异步消费。前端通过 `/api/v1/jobs/{job_id}/stream` SSE 端点订阅进度。

### 5.2 _try_extract_or_fallback()

### 5.3 递进切分：_split_recursive()

### 6.5 LLM Rerank

[reranker.py:1-129](knowledge_base_system/llm/reranker.py#L1-L129)

**重大变更**：改为批量打包打分策略，一次 LLM 调用返回所有候选评分，替代 N 次独立调用 + ThreadPoolExecutor：

```
rerank(query, candidates)
│
├── candidates 为空 → return []
│
├── candidates = 1:
│   └── _score_one(query, chunk)（81-100行）→ 单条 LLM 调用
│
├── candidates ≥ 2:
│   └── _score_batch(query, chunks)（23-78行）→ 一次 LLM 调用批量打分
│       ├── 使用 model=settings.llm_fast_model（快速模型）
│       ├── 任一条目评分失败 → 该条目不设 relevance_score，由 pipeline 层回退 RRF 分
│       └── 全部失败 → 所有条目回退，不含 relevance_score
│
├── 代码层按 relevance_score 降序排列（128行）
│
└── _score_one() / _score_batch():
    ├── build_rerank_message / build_rerank_batch_message → LLM prompt
    ├── llm_client.chat_json(messages, schema, model=llm_fast_model)
    ├── 成功 → 返回 {chunk_id, relevance_score: 0~1, reason}
    └── 失败 → 返回 {chunk_id, reason: "..."}（不含 relevance_score）
```

### 13.3 解析器链接分类

[parsers/utils.py:114-135](knowledge_base_system/parsers/utils.py#L114-L135) — `classify_link_text()`

| 链接文字示例 | 分类结果 |
|-------------|----------|
| `天空.png`、`photo.jpg` | `image_link` |
| `演示.mp4`、`video.mov` | `video_link` |
| `手册.pdf`、`报告.docx` | `document_link` |
| `百度`、`了解更多` | `web_link` |

> 占位符格式统一为 `{{type:N}}`（如 `{{image:1}}`、`{{doc:2}}`、`{{web:3}}`），由各解析器的 `next_ph()` / `next_placeholder()` 方法生成。
