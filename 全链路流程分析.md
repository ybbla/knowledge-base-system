### 3.2 POST /documents/upload — 上传文档

[documents.py:290-518](knowledge_base_system/app/api/v1/documents.py#L290-L518) — `upload_document()` + `_do_upload()`

```
upload_document(file, title, category, replace_doc_id, confirm_replace)
│
├── [1] 读取文件内容 → 计算 source_hash = sha256(content) → 获取 size
│
├── [2] 重复内容检测（323-339行）
│   └── document_repo.find_by_hash(source_hash)
│       ├── 命中 → 返回 {"duplicate": True, "existing_doc_id": ..., ...}
│       └── 未命中 → 继续
│
├── [3] 同名文档检测（342-358行）
│   └── 条件: document_repo 可用 AND replace_doc_id 为空
│       └── document_repo.find_similar_by_filename(original_name)
│           ├── 仅一个同名文档 AND confirm_replace=False → 返回 {"suggested_replace": True, ...}
│           └── 无同名或已确认替换 → 继续
│
├── [4] 替换场景校验（361-381行）
│   └── 条件: replace_doc_id 非空 AND confirm_replace=True AND document_repo 可用
│       ├── document_repo.get(replace_doc_id)
│       │   ├── 不存在 → 404
│       │   └── status != active → 409 "状态不是 active，无法替换"
│       ├── title/category 未填 → 沿用旧文档的值
│       └── 记录 old_doc + replace_doc_id_typed（供热路径使用）
│
├── [5] 提交到 upload 线程池（383-397行）
│   └── asyncio.get_running_loop().run_in_executor(upload_executor, _do_upload, ...)
│       └── 热路径在 kb-upload 线程池中同步执行，不阻塞事件循环
│
└── _do_upload() 热路径（405-518行）:
    │
    ├── [5a] 创建 Document 预占位（422-441行）
    │   ├── new_id("doc") → pre_doc_id
    │   ├── Document(source_uri="" 先为空，等文件写完后更新)
    │   │   └── previous_doc_id = replace_doc_id（替换场景记录来源）
    │   ├── document_repo.create(doc)
    │   │   ├── 成功 → 继续
    │   │   └── DuplicateDocumentError → 409 冲突（数据库唯一索引防竞态）
    │   └── 设计意图：先占位再写文件，防止并发重复上传产生孤儿文件
    │
    ├── [5b] 写文件到 MinIO（444-466行）
    │   └── upload_api.save_upload_file(file_content, original_name, size, ...)
    │       ├── 上传到 MinIO kb-input bucket → 返回 {"source_uri": "minio://...", ...}
    │       └── 失败 → document_repo.hard_delete(pre_doc_id) 回滚预占位（物理删除）→ 500
    │
    ├── [5c] 更新 source_uri（469-474行）
    │   └── doc.source_uri = upload_data["source_uri"]
    │       └── document_repo.update(doc) → 将真实 MinIO 路径写回 PG
    │
    ├── [5d] 软删除旧文档（替换场景，MinIO 成功后执行）（477-492行）
    │   └── 条件: old_doc 非空
    │       ├── document_repo.soft_delete(old_doc.doc_id)
    │       ├── chunk_store.bulk_update_status_by_doc_id(old_doc_id, "deleted")
    │       └── 逐条 sync_index_metadata(chunk) → 同步索引状态
    │
    ├── [5e] 入库（必定执行，失败标记 failed 不抛异常）（495-508行）
    │   └── ingestion_pipeline.ingest(doc, raw_content=file_content)
    │       ├── 成功 → response_data = _doc_to_item(doc)
    │       └── 失败 → doc.status = failed, response_data["ingest_error"] = True
    │
    └── [5f] 返回 Document 详情
        └── response_data 包含 duplicate/suggested_replace/replaced/replaced_doc_id 等标记
```

```

│

### 4.2 旧数据清理：_cleanup_previous_artifacts()

[pipeline.py:150-154](knowledge_base_system/ingestion/pipeline.py#L150-L154)

按依赖顺序分三步清理（chunks → elements → assets）：

```
_cleanup_previous_artifacts(doc_id)
│
├── _cleanup_old_chunks(doc_id)（123-136行）
│   ├── chunk_store.list_by_doc_id(doc_id) → 获取旧知识块

├── 遍历 vector_results（已按分数降序）
│   └── rank 从 1 开始: score += 1/(k + rank)
│
├── 遍历 bm25_results（已按分数降序）
│   └── rank 从 1 开始: score += 1/(k + rank)（累加到已有分数）
│
└── 返回 {chunk_id: fused_score}
```

### 6.5 LLM Rerank

[reranker.py:45-77](knowledge_base_system/llm/reranker.py#L45-L77)

```
rerank(query, candidates)
│
├── candidates 为空 → return []
│
├── candidates = 1:
│   └── 串行 _score_one(query, chunk)
│
├── candidates > 1:
│   └── ThreadPoolExecutor(max_workers=min(15, len(candidates)))
│       └── as_completed 收集结果
│
├── 按 relevance_score 降序排列
│
└── _score_one(query, chunk)（21-42行）:
    ├── build_rerank_message(query, chunk.content) → LLM prompt
    ├── llm_client.chat_json(messages, schema=RERANK_SCHEMA) → {relevance_score: 0~1, reason: "..."}
    ├── 成功 → 返回 {chunk_id, relevance_score, reason}
    └── LLMError → 返回 {chunk_id, reason: "LLM 调用失败，使用 RRF 融合分数"}
        └── 不含 relevance_score，pipeline 层从 score_map 取 RRF 分回退
```
